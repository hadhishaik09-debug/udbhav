Build a modern, scalable mobile application frontend named Aushadhara, designed as a medical emergency and medicine access platform. The application should be developed using React Native with Expo and must focus only on the frontend UI, screens, navigation flow, and modular structure, without implementing backend services, APIs, or databases. Mock data or placeholders can be used wherever necessary. When the application launches, it should first display a Splash Screen showing the Aushadhara logo and name, then navigate to a common Authentication Screen containing both registration and login options. The registration screen must include fields for full name, email, address, phone number, password, and confirm password, while the login screen must include email, password, and a forgot password option. After login, the user should be directed to a Main Dashboard for role selection that displays three cards: User, Pharmacy, and Delivery, and selecting each role should open the corresponding role-based interface.

The User module should allow users to log in directly and access the User Dashboard, which includes multiple features. It should contain an AI Health Assistant designed as a chat-style interface where users can enter symptoms through text or voice input. The assistant should display follow-up questions and responses in chat bubbles and include a visible severity indicator showing mild, moderate, or severe conditions. Based on severity, the interface should show suggestions such as home remedies, over-the-counter medicines, monitoring advice, or emergency alerts recommending hospital visits. AI responses should appear as text and also include a voice playback option. The dashboard should also include a voice-to-text medicine search feature where users can tap a microphone button, speak the medicine name, and view results. The Medicine Search Results screen should display nearby pharmacies showing medicine availability, price, distance, and address with sorting options like nearest pharmacy, lowest price, and stock availability. Each pharmacy card should include options to view the pharmacy on a map and add the medicine to the cart.

The application must also include map integration where pharmacies appear as location pins on a map showing their distance and address when a user searches for medicines. The booking and ordering flow should include a cart screen where users can review medicines, adjust quantities, or remove items, followed by a checkout screen where they can enter a delivery address and choose between home delivery or takeaway. An order confirmation screen should display the order summary, pharmacy details, delivery information, and a map preview of the pharmacy location.

Inside the user dashboard there should also be a Medicine Reminder module where users can create reminders for medicines. Users should be able to enter the medicine name, select reminder time, and specify dosage type such as half tablet, full tablet, or syrup, along with the quantity or dose amount. The reminder interface should clearly display the time of reminder and the dosage instruction, and each reminder card should include actions like snooze, skip, and mark as taken, along with a notification-style reminder UI.

The Pharmacy module must include a verification-based registration and login system. Pharmacies should not be able to log in immediately after registering. During registration, pharmacies must provide details such as pharmacy name, owner name, address, contact number, license number, and upload a pharmacy license image. After submission, the interface should show a “Pending Verification” status indicating that the pharmacy is waiting for admin approval. The admin interface should allow reviewing the pharmacy license details and approving or rejecting the registration. Only after approval should the pharmacy be able to log in and access the Pharmacy Dashboard, where they can manage inventory, add medicines, update stock quantities, edit medicine prices, and accept or reject user orders.

The Delivery module should follow a similar verification flow. Delivery partners must register by providing name, phone number, ID proof upload, and vehicle type. After registration, their account should remain in a pending approval state until verified by the admin. Once approved, delivery partners should be able to log in and access the Delivery Dashboard, where they can view assigned orders, accept delivery tasks, and update delivery status to picked up, out for delivery, or delivered, along with an online/offline availability toggle.

An Admin interface UI should also be designed to manage verification and approvals. The admin should be able to view pending pharmacy registrations, verify pharmacy license details, approve or reject pharmacies, view delivery partner registration requests, and approve or reject delivery accounts. The entire frontend should follow a clean medical design theme using soft healthcare colors like white, blue, and green, with card-based layouts, clear typography, icons for major features, reusable UI components, and smooth navigation. Navigation between screens should be handled using React Navigation, while the UI components should follow a modular structure suitable for a scalable production-level application. The final result should represent a fully structured frontend prototype of the Aushadhara mobile application built using React Native Expo, implementing all screens, navigation, and UI interactions without backend functionality.
